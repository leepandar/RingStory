## 业务设计和产品需求审查报告

### 1. 业务需求设计问题

#### 1.1 功能缺陷与漏洞（P0/P1优先级）

| 问题                                   | 影响                                                         | 建议                                                         |
| :------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **邀请链接二维码生成机制缺失**         | 产品文档要求"生成带参数的小程序码"，但技术文档无实现细节；未定义如何从链接生成小程序码 | 需补充：小程序码生成API对接（微信官方API），码的有效期管理，码图片的存储路径 |
| **照片删除时关联数据清理规则不明确**   | 删除照片时涉及：备注版本历史、人脸标注、评论、点赞、标签关联、影集关联等7个表；文档未定义删除顺序和冲突处理 | 需补充：删除顺序、级联删除vs软删除的混用策略、已生成回顾中的关联照片失效处理 |
| **年轮图谱"稀疏月份合并"算法未实现**   | 文档4.6.3定义："若某个月份的照片少于3张，自动将相邻月份合并为'季度碎片'节点" | 实现上存在问题：1) 合并逻辑如何定义相邻性 2) 跨季度合并时如何展示 3) 无后端代码实现 |
| **搜索功能缺少关键字段**               | 搜索接口支持关键词、人物、地点、时间筛选，但未实现：1) 多条件组合的优先级排序 2) 搜索结果分页标准 3) 模糊搜索阈值 | 需补充：搜索API分页参数、模糊匹配容差值、多条件权重配置      |
| **年轮放映室降级方案不完善**           | 文档4.8.3提到"若第三方服务不可用，回退到生成拼图海报"，但未定义：1) 降级时用户是否被通知 2) 重试机制 3) 降级后的用户体验差异 | 需补充：降级检测规则、降级通知文案、自动重试策略（延迟多久、重试几次） |
| **多人同时编辑备注的冲突处理过于简化** | 文档4.7.4定义冲突处理："后保存者会看到'备注已被他人修改，请刷新后重试'"，但这对备注内容可能丢失 | 改进方案：采用Operational Transformation或CRDT算法支持协作编辑，或采用乐观锁+版本控制 |

#### 1.2 用户体验设计问题

| 问题                                     | 影响                                                         | 建议                                                         |
| :--------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **"只读成员"权限过于严格**               | 文档2.3定义只读成员"仅允许查看和评论，不允许上传或删除"，但未考虑：长辈想为照片添加备注、语音记录等参与形式 | 改进方案：引入"只读成员可补充备注但无编辑权"的角色，丰富参与方式 |
| **邀请链接过期后的恢复流程低效**         | 过期后提示"联系管理员重新邀请"，需要两步操作；无自动重新生成选项 | 改进方案：提供"申请邀请"按钮，管理员收到通知后一键重新生成   |
| **照片上传时EXIF解析失败的处理不够友好** | 文档4.5.1规定："若无法读取EXIF，则使用文件修改时间；若异常则默认使用当前时间，提醒用户手动修改" | 改进方案：1) 上传后在详情页突出显示"拍摄时间不准确，请修正" 2) 提供日期选择器快速修改 3) 支持批量修改同一天的多张照片 |
| **年轮放映室"季度之星"选择算法不够公平** | 文档4.8.1：按"人脸出现频率最高"选择季度之星，但未定义：多个人频率相同时的排序规则 | 改进方案：补充规则："若频率相同，则按首次出现时间"或"用户操作反馈（如主动标记）" |
| **搜索历史记录清理策略缺失**             | 文档4.9.2：展示"最近10条搜索历史"，但未定义保留时间和清理时机 | 补充规则：保留30天或100条记录，用户手动清空或隐私模式下不记录 |

#### 1.3 商业化与社区管理缺陷

| 问题                           | 影响                                                         | 建议                                                         |
| :----------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **存储空间上限的触发点不一致** | 免费版上限1000张，家庭版10万张，但未定义：照片大小如何计算（原图vs压缩图）、HDR/4K照片如何处理 | 补充规则：1) 按压缩后大小计算 2) 为超大分辨率照片设定特殊处理（如自动限制分辨率） 3) 明确计算公式在用户端展示 |
| **付费转化触发不明确**         | 文档1.5："存储达到10万张后开启付费"，但实际触发机制未定义：是否允许临时超限、超限后是否禁用上传功能 | 补充规则：1) 达到90%时提示升级 2) 达到100%时禁用上传但允许浏览 3) 提供3天试用期 |
| **管理员责任边界模糊**         | 照片违规拦截、存储超限、成员移除等管理员通知，但未定义：管理员的必需操作、处理超时后的自动化降级 | 补充流程：1) 违规照片管理员必须在7天内处理 2) 超时则自动删除 3) 存储超限提醒后30天未处理则限制所有成员上传 |

### 2. 数据模型设计问题

#### 2.1 核心表结构问题

| 表名                 | 问题                                                         | 建议                                                         |
| :------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **t_photo**          | 1) `shoot_time`字段未设置NOT NULL，导致时间轴排序结果不稳定；2) 年轮图谱分表依赖`shoot_time`，但允许NULL会破坏分表策略 | 修改：`shoot_time DATETIME NOT NULL`，若EXIF无效则默认为`upload_time` |
| **t_photo**          | 3) `is_hidden_location`字段为TINYINT(0/1)，全局隐藏位置需读取`t_family.global_hide_location`后再计算，逻辑分散 | 改进：新增冗余字段`effective_is_hidden_location`，由后端计算填充，避免关联查询 |
| **t_photo**          | 4) `like_count`和`comment_count`为冗余统计字段，但未定义更新时机和一致性保证 | 规范：1) 由RocketMQ消费者异步更新，不保证强一致性 2) 缓存与数据库定期对账（每小时） |
| **t_family_member**  | 1) 缺少`role`字段的"只读"角色定义，现有枚举仅支持"admin/member"，与文档不符 | 补充：`role` ENUM('admin', 'member', 'viewer')，映射文档的"只读成员" |
| **t_family_member**  | 2) `status`字段区分"已移除"和"正常"，但软删除后`deleted_at`与`status`含义重复 | 简化：删除`status`字段，仅用`deleted_at`标记逻辑删除；移除成员改为标记`deleted_at` |
| **t_photo_note**     | 1) 版本控制依赖单独的`t_photo_note_history`表，但未在`t_photo_note`中记录`version`号 | 改进：在`t_photo_note`中增加`current_version INT`字段，指向最新版本，便于快速查询 |
| **t_photo_note**     | 2) `mentioned_users`字段存储为VARCHAR(512)的逗号分隔字符串，扩展性差；难以支持"快速查询@某用户的所有备注" | 改进：新建关联表`t_photo_note_mention`记录被@的用户，便于索引查询 |
| **t_moments_review** | 1) `photo_ids`存储为VARCHAR(1024)的逗号分隔，无法支持"修改某张照片后自动更新已生成的回顾列表" | 改进：新建关联表`t_review_photo`记录回顾包含的照片，便于级联更新 |
| **t_moments_review** | 2) 缺少`duration`字段（视频/海报的时长），客户端无法预知播放时间 | 补充：`duration_seconds INT COMMENT '时长（秒）'`            |
| **t_face_photo**     | 1) `embedding`字段存储为TEXT格式的"逗号分隔512维向量"，无法在ES中支持向量检索 | 改进：向量数据下沉至Elasticsearch中的`dense_vector`字段，MySQL仅保存`face_id`引用 |
| **t_photo_album**    | 缺少`sort_order`字段的排序规则定义（ASC/DESC），导致客户端排序结果不确定 | 补充：规范为`sort_order ASC`，新增照片时自动计算最大值+1     |

#### 2.2 索引设计问题

| 表名               | 现有索引                                                     | 问题                                                         | 建议                                                         |
| :----------------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **t_photo**        | `idx_shoot_time`, `idx_family_id` 分别独立                   | 年轮图谱查询条件为`family_id + shoot_time`，应建立联合索引；当前分别独立，MySQL可能选择错误执行计划 | 新增：`idx_family_shoot_time(family_id, shoot_time DESC)` 便于倒序查询 |
| **t_comment**      | 仅有`idx_photo_id`, `idx_parent_id`                          | 查询某用户的评论需遍历全表，无`idx_author_id`索引            | 新增：`idx_author_id(author_id, create_time DESC)`           |
| **t_like**         | 有`uk_photo_user`唯一约束                                    | 无反向查询"某用户点赞的所有照片"的索引                       | 新增：`idx_user_photo(user_id, photo_id)`                    |
| **t_notification** | `idx_recipient_id` 包含`is_read`，但查询通常按`create_time`倒序 | 无`create_time`索引，分页查询效率低                          | 优化：`idx_recipient_read_time(recipient_id, is_read, create_time DESC)` |
| **t_photo_note**   | 无`family_id`索引                                            | 某些查询需关联`t_photo`表后按`family_id`筛选，无直接索引     | 新增：`idx_family_author(family_id, author_id)` 便于快速查询某成员的所有备注 |

#### 2.3 数据一致性风险

| 风险                                 | 根源                                                         | 影响                                | 建议                                                         |
| :----------------------------------- | :----------------------------------------------------------- | :---------------------------------- | :----------------------------------------------------------- |
| **冗余统计字段不一致**               | `t_photo.like_count`, `t_family.photo_count`, `t_family.member_count`为冗余字段，由不同的异步消费者更新 | 年轮图谱显示照片计数错误，用户疑惑  | 1) 建立定时对账任务（每小时） 2) 关键业务逻辑避免依赖冗余字段，改为实时COUNT |
| **软删除与业务逻辑混乱**             | 软删除字段`deleted_at`与逻辑删除的关系未明确；某些查询需额外加`WHERE deleted_at IS NULL`条件 | 容易误查出已删除数据；代码容易出bug | 1) 使用MyBatis-Plus的`@TableLogic`注解统一管理 2) 业务查询默认自动过滤删除数据 |
| **邀请链接状态与实际加入状态不一致** | `t_invitation.status`记录为"已使用"，但`t_family_member`中的成员可能被移除；邀请列表显示"已加入"但成员已不存在 | 邀请历史不可靠                      | 补充规则：邀请状态改为"pending/used/expired/revoked"，成员移除时不改邀请状态 |
| **年轮图谱缓存失效机制缺失**         | 年轮图谱树结构存入Redis，但照片删除/修改时无缓存失效规则     | 用户看到的年轮图可能包含已删除照片  | 补充机制：1) 照片操作后发送RocketMQ事件 2) ringtree-svc消费事件并清除Redis缓存 3) 下次查询时重新计算 |

### 3. API接口设计问题

#### 3.1 RESTful规范违反

| 接口                                    | 现状                      | 违反规范                        | 改进建议                                                     |
| :-------------------------------------- | :------------------------ | :------------------------------ | :----------------------------------------------------------- |
| `POST /api/family`                      | 创建家庭，传递name/userId | 正确                            | 无                                                           |
| `POST /api/family/{id}/invitation`      | 生成邀请                  | 正确                            | 无                                                           |
| `POST /api/family/join`                 | 通过邀请加入家庭          | 非幂等；应使用PUT或PATCH        | 改为`PUT /api/family/{familyId}/members/{userId}?token=xxx`  |
| `PutMapping("/member/{memberId}/role")` | 更新成员角色              | 缺少更新的新值，参数分散        | 改为`PATCH /api/family/members/{memberId}` + Body: `{"role":"member"}` |
| `DeleteMapping("/member/{memberId}")`   | 移除成员                  | 正确                            | 补充：`?deletePhotos=true/false`查询参数明确是否删除成员的照片 |
| `PostMapping("/tag")`                   | 创建标签                  | 正确                            | 无                                                           |
| `PostMapping("/photo/{photoId}/tags")`  | 添加标签（可能添加多个）  | 正确但名词单数                  | 改为更明确的名称表示复数操作                                 |

| `GetMapping("/{photoId}/like/check")`   | 检查是否已点赞            | 查询用`GET`正确，但路径冗长     | 改为`GET /api/album/photos/{photoId}/like?userId=xxx`，响应包含布尔值 |
| `PostMapping("/album")`                 | 创建影集                  | 正确                            | 无                                                           |

#### 3.4 错误响应与边界情况处理

### 4. 技术架构与数据模型不匹配问题

#### 4.1 分库分表策略问题

| 问题                              | 影响                                                         | 建议                                                         |
| :-------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **t_photo按shoot_time年份分表**   | 跨年查询（如"去年春节到今年元旦"）需查询多张表，且ShardingSphere需配置多路由 | 1) 建立清晰的年份到表名映射表 2) 查询时自动计算涉及的表名列表 3) 支持异步merge结果集 |
| **分库键为family_id MOD 4**       | 大家庭（10万张照片）会集中在某个分库，导致热点问题           | 监控各分库的存储和查询压力，若某个家庭单库数据超过10GB则考虑二级分片（按`uploader_id`） |
| **soft delete后分表字段可能变化** | 若照片被逻辑删除，`deleted_at`被赋值，但分表字段`shoot_time`不变，导致已删除照片仍占用分表空间 | 补充：定期清理任务，180天后的已删除照片物理删除              |
| **跨分库关联查询低效**            | `t_photo JOIN t_photo_note`在分库场景下需进行分布式JOIN，效率低 | 优化：在`t_photo`中冗余`note_count`字段，避免跨表关联        |

#### 4.2 缓存与数据库不一致

| 问题                  | 根源                                                  | 建议                                                         |
| :-------------------- | :---------------------------------------------------- | :----------------------------------------------------------- |
| **年轮图谱Redis缓存** | 存储树结构，但照片新增/删除时无主动失效机制           | 补充：1) RocketMQ事件驱动 2) 设定TTL自动过期 3) 使用主从版本号检测失效 |
| **点赞/评论计数缓存** | `t_photo.like_count`与Redis中的计数不一致             | 采用最终一致性方案：1) RocketMQ消费事件更新Redis 2) 后端定期对账 3) 3分钟内允许误差 |
| **成员列表缓存**      | `t_family_member`的列表被缓存，但成员移除后缓存未失效 | 补充：成员操作事件发送后主动清除缓存                         |

#### 4.3 搜索引擎(Elasticsearch)使用问题

| 问题                         | 影响                                                         | 建议                                                         |
| :--------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **ES索引设计缺少必要字段**   | 缺少`created_at`字段，无法按创建时间排序搜索结果             | 补充字段：`created_at`, `updated_at`, `is_favorite`          |
| **人脸特征向量存储位置混乱** | 设计文档说"向量存入ES"，但数据库设计又在`t_face_photo.embedding`中存储 | 明确：向量数据主要存储在ES的`dense_vector`字段中，MySQL仅保留reference；检索时优先查ES |
| **多语言搜索支持缺失**       | 备注内容支持markdown富文本，但ES分析器仅配置了`ik_smart`中文分词器 | 补充：支持中英文混合搜索，配置`ik_smart` + `standard`混合分析器 |
| **搜索结果的实时性保证不足** | 新上传照片到出现在搜索结果的延迟未定义                       | 规范：1) 照片上传后立即同步至ES（不超过1秒） 2) 失败则记入死信队列并重试 |

### 5. 业务流程的逻辑漏洞

#### 5.1 照片生命周期管理

| 阶段          | 问题                                                         | 建议                                                         |
| :------------ | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **上传前**    | 无预检查：是否超过存储上限、文件格式是否支持、文件大小是否过大 | 补充：1) 前端上传前检查剩余空间 2) 后端再次验证 3) 超限时返回详细错误码 |
| **上传中**    | 分片上传失败后的重试机制不清；网络中断后是否能恢复           | 规范：1) 分片上传使用MD5校验 2) 支持断点续传 3) 相同uploadId + partNumber的重复请求幂等 |
| **上传后**    | EXIF解析和异步处理任务的顺序不清：是否等待全部任务完成才标记为"正常"状态 | 规范：1) 上传后立即标记为"处理中" 2) 内容审核通过后标记为"正常" 3) 违规则标记为"拦截" 4) 检测完成无论通过否都可查看（但违规需管理员操作） |
| **删除照片**  | 删除时是否检查该照片是否在回顾中被引用；删除后用户已分享的回顾链接是否失效 | 规范：1) 删除前查询是否在已生成的回顾中 2) 若存在则提示"该照片已包含在某些回顾中" 3) 允许删除但回顾显示占位符 |
| **导出/备份** | 无明确的导出策略：是否包括已删除照片、评论、备注、标签等关联数据 | 补充：导出选项面板：1) 包含/排除已删除照片 2) 包含/排除评论和备注 3) 导出格式（zip/JSON + photos） |

#### 5.2 成员权限管理漏洞

| 场景                       | 问题                                                         | 建议                                                         |
| :------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **成员被移除后**           | 被移除成员的照片、备注、评论是否被级联删除、归档或保留？未定义 | 规范：1) 照片保留但标记上传者为"已移除用户" 2) 备注和评论保留 3) 用户账户保留但不显示在成员列表 |
| **管理员权限转让**         | 文档允许"转让家庭所有权"，但未定义：原管理员是否保留其他权限，是否可同时存在多个管理员 | 改进：1) 明确规范"只读成员/成员/管理员"三级权限模型 2) 支持多管理员 3) 转让时原管理员自动降级为成员 |
| **只读成员的补充能力不足** | "只读成员"无法为照片补充备注，在整体传递家庭故事时被动       | 改进：增加"受限成员"角色，可备注但无删除权限                 |

#### 5.3 邀请链接逻辑漏洞

| 阶段         | 问题                                                         | 建议                                                         |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **生成邀请** | 无限制生成邀请，可能产生过多未使用的邀请链接                 | 规范：1) 每个家庭最多同时存在10个待使用邀请 2) 管理员生成新邀请前可选择是否失效旧邀请 |
| **使用邀请** | 邀请链接可能被"多个人"使用（若`max_uses > 1`），但未定义如何区分 | 规范：1) 若`max_uses=1`则仅一人可用 2) 若`max_uses>1`则多人共享链接，每人加入时生成新的family_member记录 3) 使用次数超过上限则自动失效 |
| **邀请过期** | 7天有效期硬编码，不支持管理员自定义                          | 改进：管理员创建邀请时可设置有效期（1-30天）                 |

| **数据模型合理性** | 7/10 | 核心字段NOT NULL约束缺失、冗余字段管理混乱、索引不完整 |

2. 建立接口规范标准并通过代码生成工具（如OpenAPI/Swagger）统一维护
3. 补充数据库设计文档中的一致性保证和监控指标
